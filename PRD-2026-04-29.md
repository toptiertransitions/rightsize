# 1. System Overview

The Top Tier Transitions platform consists of two distinct applications that work together as a unified commerce and operations system:

**Rightsize by Top Tier Transitions** (`app.toptiertransitions.com`) is the internal B2B SaaS operations platform. It manages the full lifecycle of senior rightsizing projects: item cataloging, AI-assisted valuation, routing to sales channels, vendor coordination, contracts, invoicing, payroll, CRM, and an admin backend. It is multi-tenant, meaning each client household operates as its own workspace.

**ProFound Finds** (`profoundfinds.com`) is the public-facing luxury consignment storefront. It reads live inventory from Rightsize via authenticated API calls and records completed sales back to Rightsize. It has no authentication of its own — all shoppers are anonymous. It supports both standard consignment listings and Dutch auction estate sales.

The two apps communicate as follows:
- ProFound Finds fetches items and estates from Rightsize via `GET /api/storefront/*` (authenticated with `x-storefront-api-key` header)
- ProFound Finds reserves, unreserves, and sells items in Rightsize via `POST /api/storefront/items/[id]/reserve`, `POST /api/storefront/items/[id]/unreserve`, and `POST /api/storefront/sale`
- Rightsize notifies ProFound Finds of item changes via `POST` to `STOREFRONT_URL/api/storefront/webhook/item-updated` (HMAC-signed with `STOREFRONT_WEBHOOK_SECRET`)

# 2. Application 1: Rightsize by Top Tier Transitions

## 2.1 Business Overview

Rightsize is a multi-tenant B2B SaaS platform for Top Tier Transitions (TTT), a senior downsizing and relocation services company based in the Chicago area. The platform manages the full lifecycle of rightsizing projects:

- Cataloging household items (with AI-assisted photo analysis)
- Routing items to various sales channels (ProFoundFinds consignment, Facebook Marketplace, eBay, Square POS in-person, estate sales, donation, discard)
- Coordinating vendor relationships (movers, realtors, donation orgs, consignment stores, etc.)
- Generating contracts and invoices (with e-signature and QuickBooks sync)
- Tracking staff time, mileage, and payroll
- Managing a CRM and lead pipeline (with Gmail and Google Calendar integration)
- Running email drip campaigns for referral partner outreach

Each client project (household) is a "tenant" with its own workspace, users, items, rooms, and contracts. TTT staff have elevated roles that span all tenants.

**Production URL:** `https://app.toptiertransitions.com`
**Vercel project:** Deployed on Vercel

## 2.2 User Roles

The system uses a hybrid role model: per-tenant membership roles for client access, and global TTT staff roles for internal operations.

| Role | Description |
|---|---|
| `Owner` | The client who owns the household being rightsized. Full edit access within their own tenant. |
| `Collaborator` | A family member or helper added by the Owner. Edit access within the tenant. |
| `Viewer` | Read-only project access within a tenant. |
| `TTTStaff` | Top Tier Transitions field staff. Can view and edit items, log time, access plan/schedule, and post internal notes on the /plan page. |
| `TTTManager` | TTT manager. Full access including payroll admin, all admin routes, editing "Not Included in this Scope" in contracts, and re-printing payout PDFs. |
| `TTTSales` | TTT sales and CRM staff. Access to CRM, opportunities, drip campaigns, and internal notes board. |
| `TTTAdmin` | Full system administrator. Access to all admin routes, impersonation, migrations, re-printing payout PDFs. |

**Role assignment mechanism:**
- `TTTAdmin` and `TTTManager` roles are controlled via the `TTT_ADMIN_USER_IDS` environment variable (comma-separated Clerk user IDs).
- `TTTStaff` roles are controlled via the `TTT_STAFF_USER_IDS` environment variable.
- Per-tenant `Owner`, `Collaborator`, and `Viewer` roles are stored in the Memberships Airtable table.
- Role-check helper functions: `isTTTAdmin(userId)`, `isTTTStaff(userId)`, `isTTTManager(userId)`.

**Role capability summary (as of April 29, 2026):**

| Feature | Owner/Collaborator/Viewer | TTTStaff | TTTManager | TTTSales | TTTAdmin |
|---|---|---|---|---|---|
| View contract (with "Not Included" section) | View-only | — | Edit + View | — | Edit + View |
| Internal Notes board (TTT-only) | Never visible | Read + Post | Read + Post | Read + Post | Read + Post |
| Search catalog by Estate Sale ID | No | No | Yes | No | Yes |
| Re-print client payout PDF | No | No | Yes | No | Yes |

## 2.7 Key Business Logic

### 2.7.1 Calculator Engine (Public `/calculator` Page)

The calculator estimates total project hours for a rightsizing project based on room inventory.

**Base rate:** 1 hour per 100 square feet of living space.

**Density multipliers** applied to the base rate per room:
- `Low` density: 0.5× multiplier
- `Medium` density: 1.0× multiplier
- `High` density: 2.0× multiplier

**Helper capacity** (maximum hours per week the team can deliver):
- Solo client (no help): 10 hours/week
- Family helpers: 20 hours/week
- Mixed (family + some TTT): 35 hours/week
- TTT full team: 50 hours/week

**Additional services:**
- Packing: 1.0 hour per 100 square feet of the destination space
- Unpacking: 1.0 hour per 100 square feet of the destination space

**Process:** Iterate over all rooms, multiply `(sqft / 100) × densityMultiplier` to get rightsizing hours, then add packing/unpacking hours. Divide total hours by helper capacity to estimate duration in weeks.

### 2.7.3 Dutch Auction Pricing

Used for Online estate sales. The price drops automatically on a schedule.

**Formula:**
```
n = floor((now - saleStartMs) / (dropIntervalHours × 3600000))
floorPrice = startingPrice × (floorPercent / 100)
currentPrice = max(startingPrice × (1 - dropPercent/100)^n, floorPrice)
nextDropAt = new Date(saleStartMs + (n + 1) × dropIntervalMs).toISOString()
atFloor = (currentPrice <= floorPrice)
```

**Parameters (from Estates record):**
- `startingPrice` = the item's `ValueMid` or explicitly set listing price
- `dropIntervalHours` = hours between each price drop
- `dropPercent` = percentage to reduce price at each drop (e.g., 10 means 10% reduction per interval)
- `floorPercent` = minimum price as a percentage of starting price (e.g., 50 means price never goes below 50% of start)
- `saleStart` = `estate.SaleStartDate`

**Returns:** `{ currentPrice, startingPrice, nextDropAt, atFloor }`

This calculation runs in the Rightsize API at `lib/estate-utils.ts`. It is also replicated server-side in ProFoundFinds' `POST /api/storefront/stripe/create-intent` to prevent price manipulation.

### 2.7.4 Payroll Calculations

All payroll calculations are performed in the `/api/admin/pay/*` routes.

**Hourly Pay:**
```
pay = (totalMinutes / 60) × hourlyRate
```
Where `hourlyRate` comes from `StaffRoles.HourlyRate` for the staff member.

**Commission:**
```
commission = salePrice × (staffCommissionPercent / 100)
```
Applied to all Items where `Status = Sold` and `StaffSellerId` is set and `CommissionPaidAt` is null (unpaid).

**Mileage Reimbursement:**
```
reimbursableMiles = max(0, totalMilesForDay - 20)
pay = reimbursableMiles × mileageRate
```
Grouped by `ClerkUserId + Date`. The first 20 miles per staff member per day are not reimbursed (considered personal commute). `mileageRate` comes from `ContractSettings.MileageRate` or a global default.

**Travel Time Pay:**
```
reimbursableMinutes = max(0, minutes - 30)
pay = (reimbursableMinutes / 60) × hourlyRate
```
Applied per time entry where `TravelMiles > 0`. The first 30 minutes of each travel entry is not paid.

**Expense Reimbursement:**
Sum of all expenses where `Reimbursable = true` and `PaidAt` is null (for unpaid view) or within date range (for paid view).

### 2.7.5 Routing Rules Engine

The routing rules engine automatically assigns `PrimaryRoute` and optionally a vendor type to items based on configurable rules stored in the `RoutingRules` table.

**Evaluation order:** Rules are sorted by `Priority` (ascending — lower number = higher priority). The first rule that matches an item wins; subsequent rules are not evaluated for that item.

**Matching criteria (all provided filters must match):**
- `ConditionFilter`: Item's `Condition` must be in the filter array (if filter is set)
- `CategoryFilter`: Item's `Category` must be in the filter array (if filter is set)
- `MinValue` / `MaxValue`: Item's `ValueMid` must fall within the range (if set)
- `SizeFilter`: Item's `SizeClass` must be in the filter array (if set)

**Result:** Matching item is assigned `PrimaryRoute` and optionally the `VendorType` specified in the rule.

**Applied via:** `POST /api/admin/routing-rules/apply` — runs all active rules against all items in a tenant that do not yet have a route, or against all items if requested.

### 2.7.6 Contract Workflow

1. Admin creates a contract from a template (HTML body with placeholder variables like `{{clientName}}`, `{{totalAmount}}`).
2. Admin adds service line items: each item specifies `{ name, hours, rate, total }`. `total = hours × rate`.
3. Admin fills in the "Not Included in this Scope" textarea (pre-filled from `settings.notInScopeDefault`). This field is stored on the contract as `notInScope`.
4. Admin can preview the contract (React-PDF rendering). PDF includes the "Not Included in this Scope" section when `notInScope` is set.
5. Admin sends contract via Resend email. The email includes a link to `https://app.toptiertransitions.com/sign/[signingToken]` where `signingToken` is a UUID stored on the contract record. Contract status → `Sent`.
6. Client visits the `/sign/[token]` public page (no auth required). The page shows a read-only contract preview (including the "Not Included in this Scope" section if set) and a signature canvas.
7. Client draws signature and submits. `POST /api/contracts/sign` validates the token, stores the signature as a base64 DataURL in `SignatureDataUrl`, sets `SignedAt` timestamp, and sets `Status = Signed`.
8. Optionally, the admin can trigger creation of a deposit invoice from the signed contract.

---

### 2.7.7 Invoice Workflow

1. Admin creates an invoice (`Type = Deposit` or `Full`).
2. Invoice contains `LineItems` (service × hours × rate) and optionally `ExpenseItems` from the Expenses table.
3. `Subtotal = sum(lineItems)`, `Tax` is calculated per settings, `Total = Subtotal + Tax`.
4. Admin sends invoice via Resend. Email includes a PDF attachment (React-PDF generated via `GET /api/invoices/[id]/pdf`). Status remains `Unpaid`.
5. When payment is received, admin records `AmountPaid` and `PaidAt`. If `AmountPaid >= Total`, status → `Paid`. If partial, status → `PartiallyPaid`.
6. Optionally, admin syncs invoice to QuickBooks Online via `POST /api/qbo/invoice`. This creates or updates a QBO Invoice and stores `QBOInvoiceId` and `QBODocNumber` back on the Rightsize record.

---

### 2.7.8 Square POS Consignment Flow

1. Item is cataloged with `PrimaryRoute = ProFoundFinds Consignment` and status moves to `Approved`.
2. Admin runs Square catalog sync (`POST /api/square/sync`). For each approved item, a `CatalogItem` is created in Square with a single `CatalogItemVariation`. The variation's SKU is the item's `BarcodeNumber`. The item record is updated with `SquareCatalogId`, `SquareVariationId`, and `LastSquareSyncAt`.
3. TTT staff uses Square POS terminal in-store. When an item is sold via Square, the Square webhook fires `POST /api/square/webhook` (public endpoint, HMAC signature validated).
4. Webhook handler creates an `ItemSaleEvent` record with order ID, payment ID, quantity, unit price, and total amount. Calculates `ClientPayoutAmount = totalAmount × (clientSharePercent / 100)`.
5. Item `Status` → `Sold`, `SalePrice` and `SaleDate` are recorded on the item.
6. Admin can run `POST /api/square/sync-sales` periodically to catch any missed webhook events.
7. Admin generates payout PDF via the Sales page. The payout PDF route supports a `viewOnly` param — when `true`, it returns `pdfBase64` and skips Cloudinary upload and Airtable write. PDFs are opened from blob URLs (no proxy needed).

---

### 2.7.9 Payout PDF and Re-print Flow

The `/sales` page (`SalesClient.tsx`) provides:

- **"Payout Client"** button: generates payout PDF for unpaid items, marks items as paid, opens PDF via blob URL from `pdfBase64`.
- **"Re-print Payout"** button (outlined green): uses `buildPaidLineItems` instead of `buildUnpaidLineItems`, calls `payout-pdf` with `viewOnly: true`, opens PDF from blob URL — skips marking items paid again and skips Cloudinary upload.

The `PayoutModal` accepts a `reprint` prop. When `reprint = true`, it builds paid line items and sets `viewOnly: true` in the PDF API call.

The `ProofOfPaymentSection` renders PDF links from stored Cloudinary URLs. PDF links use an `onClick` handler that calls `payout-pdf` with `viewOnly: true` and opens a blob URL — no `pdf-proxy` round-trip needed for these either.

### 2.7.10 Internal Notes Board

The `/plan` page renders `InternalNotesSection` at the bottom for any user with `isTTTStaffOrAbove` (TTTStaff, TTTManager, TTTSales, TTTAdmin). The section is never visible to clients (Owner, Collaborator, Viewer).

**Behavior:**

3. `InternalNotesSection` uses these to set author info when posting new notes/comments.

## 2.8 Auth and Security

**Authentication provider:** Clerk Auth v7. All session management, sign-in, sign-up, and token handling is delegated to Clerk. Server-side auth uses Clerk's `auth()` helper in Server Components and API routes.

**New user sync:** When a user signs up via Clerk, the Clerk webhook (`POST /api/webhooks/clerk`) fires and creates a corresponding record in the Airtable `Users` table. Webhook signature is validated using the `svix` library and the Clerk webhook signing secret.

**Middleware protection:** Next.js middleware (`middleware.ts`) protects all routes except the following public paths:
- `/` (marketing homepage)
- `/calculator`
- `/sign-in/*`
- `/sign-up/*`
- `/invite/*`
- `/api/invites/*`
- `/api/mcp`
- `/api/square/webhook`
- `/api/storefront/*`
- `/api/contracts/sign`
- `/sign/*`
- `/pay/*`

All other routes require a valid Clerk session. API routes additionally check role permissions.

**Role-check helpers:**
- `isTTTAdmin(userId: string): boolean` — returns true if `userId` is in `TTT_ADMIN_USER_IDS` env var
- `isTTTManager(userId: string): boolean` — returns true if `userId` is in `TTT_ADMIN_USER_IDS` (managers and admins share this env var in current implementation)
- `isTTTStaff(userId: string): boolean` — returns true if `userId` is in `TTT_STAFF_USER_IDS` env var
- Per-tenant roles are checked by querying the `Memberships` table for the user + tenant combination

**Storefront API security:** The `/api/storefront/*` routes bypass Clerk auth but validate the `x-storefront-api-key` header against `STOREFRONT_API_KEY`. Requests without a matching key return 401.

**Square webhook security:** Square webhook signature validated using Square's HMAC-SHA256 scheme via `SQUARE_WEBHOOK_SIGNATURE_KEY`.

**Internal Notes access:** The `/api/plan/notes` routes check that the user is TTTStaff/TTTManager/TTTSales/TTTAdmin before allowing access. Notes are never returned to client-role users.

# 3. Application 2: ProFound Finds

## 3.1 Business Overview

ProFound Finds (`profoundfinds.com`) is the public-facing luxury consignment storefront for Top Tier Transitions. It sells estate and rightsizing items directly to consumers online. All inventory is sourced from Rightsize's storefront API — the app has no local database of its own. Sales are recorded back to Rightsize via API calls after Stripe payment confirmation.

**Target demographic:** Affluent consumers, older demographic. The design must feel premium, refined, and trustworthy — not e-commerce generic.

**Business model:** TTT takes a portion of each sale; the remainder goes back to the client (controlled by `ClientSharePercent` on each item in Rightsize). ProFound Finds handles the consumer-facing experience: browsing, cart management with reservations, Stripe checkout, and post-purchase communication.

**Production URL:** `https://www.profoundfinds.com`
**Vercel project:** Deployed on Vercel with cron jobs enabled.

The app supports two types of sales:
1. **Standard consignment listings** — individual items with fixed prices on the `/shop` page
2. **Online estate sales** — Dutch auction events at `/onlineestatesales/[slug]` where prices automatically decrease on a schedule

## 3.5 Key Business Logic

### 3.5.1 Cart and Reservation System

The cart uses a client-side React Context (`CartContext`) with server-side reservation enforcement via Rightsize + Upstash Redis.

**Adding an item to cart:**
1. User clicks "Add to Cart" on a product page.
2. Client calls `POST /api/storefront/items/{id}/reserve`.
3. Server calls Rightsize `POST /api/storefront/items/{id}/reserve` → Airtable item `Status` changes to `Reserved`.
4. On success, server stores in Upstash Redis: `key = reservation:{itemId}`, `value = JSON({ itemId, sessionId, reservedUntil })`, TTL = 900 seconds.
5. `CartContext` adds `CartItem` to the cart array in local state with `reservedUntil = now + 15 minutes`.

**Reservation expiry:**
- `CartContext` polls every 10 seconds.
- If `reservedUntil < now` for any cart item, it is automatically removed from the cart.
- On removal, client calls Rightsize `POST /api/storefront/items/{id}/unreserve` → item reverts to `Listed`.

**Orphan cleanup:**
- The daily cron job catches any reservations that expired but were not properly unreserved (e.g., if the user closed their browser mid-reservation).

**Cart constraints:**
- Items from the same estate sale or same general category can coexist in cart.
- Cart state is stored in memory (React Context) — it is lost on page refresh. This is acceptable given the short reservation window.

### 3.5.2 Checkout Flow

**Step 1 — Contact Information:**
- Fields: Full Name, Email Address, Phone Number
- Client-side validation: all fields required, email format valid, phone format valid
- No account creation required

**Step 2 — Payment:**
1. Client submits contact info and proceeds to payment step.
2. Client-side calls `POST /api/storefront/stripe/create-intent` with `{ itemIds, buyerName, buyerEmail, buyerPhone, itemPrices }`.
3. Server-side verifications:
   - Each item must still have `Status = Reserved` in Rightsize (re-fetched)
   - For estate sale items: Dutch auction price is re-computed server-side using the live formula
   - If any price has changed by more than $0.01 from client-submitted prices: return HTTP 409 with `{ newPrice }` → client shows a confirmation modal with the updated price and asks user to confirm before proceeding
4. Calculate tax at **10.25%** (Illinois 6.25% + Cook County 1.75% + Chicago 1.25% + RTA 1.0%)
5. Create Stripe PaymentIntent: `amount = Math.round((subtotal + tax) * 100)` (cents)
6. Return `{ clientSecret, subtotal, tax, total }` to client
7. Client mounts Stripe Elements (`CardElement` or `PaymentElement`) with the `clientSecret`
8. User enters card details in Stripe iframe
9. Client calls `stripe.confirmPayment({ elements, confirmParams: { return_url: ... } })`

**Step 3 — Confirmation:**
- After `stripe.confirmPayment` resolves, show confirmation screen with order summary
- Stripe webhook fires asynchronously to record the sale and send email

### 3.5.3 Stripe Webhook: Payment Confirmed

On `payment_intent.succeeded` event:
1. Validate Stripe-Signature header using `stripe.webhooks.constructEvent()` with `STRIPE_WEBHOOK_SECRET`
2. Extract buyer metadata from PaymentIntent (`buyerName`, `buyerEmail`, itemId list)
3. For each item, call Rightsize `POST /api/storefront/sale` with: `{ itemId, buyerName, buyerEmail, salePrice, stripePaymentIntentId }`
4. Send purchase confirmation email via Resend to buyer's email address (includes item name, price paid, pickup address if applicable)
5. Delete Redis reservation key: `reservation:{itemId}`
6. Return 200 to Stripe

### 3.5.4 Dutch Auction Pricing

Detailed formula (same as Rightsize, replicated server-side for price verification):

```typescript
function computeDutchPrice(estate: StorefrontEstate, startingPrice: number): EstateStorefrontItem {
  const now = Date.now();
  const saleStartMs = new Date(estate.saleStartDate).getTime();
  const dropIntervalMs = estate.dropIntervalHours * 3600000;

  const n = Math.floor((now - saleStartMs) / dropIntervalMs);
  const floorPrice = startingPrice * (estate.floorPercent / 100);
  const computedPrice = startingPrice * Math.pow(1 - estate.dropPercent / 100, n);
  const currentPrice = Math.max(computedPrice, floorPrice);

  const nextDropAt = n >= 0
    ? new Date(saleStartMs + (n + 1) * dropIntervalMs).toISOString()
    : null;

  return {
    currentPrice: Math.round(currentPrice * 100) / 100,
    startingPrice,
    nextDropAt,
    atFloor: currentPrice <= floorPrice,
  };
}
```

**Client-side countdown timer behavior:**
- Timer re-fetches estate data every 30 seconds (normal polling interval)
- When `nextDropAt - now < 60000ms` (less than 60 seconds until next drop), timer switches to 1-second polling to ensure smooth countdown
- Timer displays: `HH:MM:SS` or `MM:SS` countdown to next price drop
- When `atFloor = true`, timer shows "Final Price — No Further Drops"

**Server-side enforcement at checkout:**
- `POST /api/storefront/stripe/create-intent` always re-computes Dutch price server-side
- If computed price differs from submitted price by more than $0.01 (rounding tolerance), returns HTTP 409
- Client shows modal: "The price for [item] has changed to $[newPrice]. Would you like to continue at the new price?" with Confirm/Cancel buttons

### 3.5.5 ISR Cache Strategy

| Resource | Revalidation | Trigger |
|---|---|---|
| Individual item pages (`/shop/[slug]`) | 5-minute ISR (`revalidate: 300`) | Rightsize item update → POST to ProFoundFinds `/api/storefront/webhook/item-updated` → `revalidatePath('/shop/' + slug)` |
| Estate pages (`/onlineestatesales/[slug]`) | 1-minute ISR (`revalidate: 60`) | Rightsize estate update → webhook trigger |
| Estate list pages | 1-minute ISR (`revalidate: 60`) | Automatic |
| Shop catalog | 5-minute ISR (`revalidate: 300`) | Automatic + webhook |

**Webhook validation:** When Rightsize calls `POST https://profoundfinds.com/api/storefront/webhook/item-updated`, it includes an HMAC-SHA256 signature in the `x-webhook-signature` header. ProFoundFinds verifies this signature using `STOREFRONT_WEBHOOK_SECRET` before calling `revalidatePath()`.

# 4. Inter-Application Integration

This section describes the complete integration contract between Rightsize and ProFoundFinds.

## 4.1 Rightsize → ProFoundFinds (Item Data Flow)

ProFoundFinds reads all inventory and estate data from Rightsize on demand (via ISR + on-request fetching).

**Authentication:** All requests from ProFoundFinds to Rightsize include the header:
```
x-storefront-api-key: <value of RIGHTSIZE_API_KEY env var>
```
This must match the `STOREFRONT_API_KEY` value set in Rightsize's environment.

**Endpoints consumed by ProFoundFinds:**

| Rightsize Route | When Called | Purpose |
|---|---|---|
| `GET /api/storefront/items` | Page build, ISR refresh | Fetch all active storefront items for `/shop` |
| `GET /api/storefront/items/[id]` | Page build, ISR refresh | Fetch individual item for `/shop/[slug]` |
| `GET /api/storefront/estates` | Page build, ISR refresh | Fetch all estates for listing pages |
| `GET /api/storefront/estates/[slug]` | Page build, ISR refresh | Fetch single estate for detail pages |
| `GET /api/storefront/items/reserved` | Daily cron | Get all reserved items for orphan cleanup |
| `POST /api/storefront/items/[id]/reserve` | User action: add to cart | Reserve item in Airtable |
| `POST /api/storefront/items/[id]/unreserve` | Cart expiry or manual remove | Revert item to Listed |
| `POST /api/storefront/sale` | After Stripe payment confirmed | Record sale, set buyer info, mark Sold |

## 4.2 Rightsize → ProFoundFinds (Item Update Webhooks)

When an item is updated in Rightsize in a way that affects the storefront (status change, price change, photos updated), Rightsize fires a webhook to invalidate the ProFoundFinds ISR cache.

**Request:**
```
POST https://www.profoundfinds.com/api/storefront/webhook/item-updated
Content-Type: application/json
x-webhook-signature: <HMAC-SHA256 of body using STOREFRONT_WEBHOOK_SECRET>

{
  "itemId": "<Airtable record ID>",
  "slug": "<OnlineListingSlug>",
  "status": "<new status>"
}
```

**ProFoundFinds response:** Calls `revalidatePath('/shop/' + slug)` and returns 200.

## 4.3 Sale Recording Flow (End-to-End)

1. Shopper adds item to cart → ProFoundFinds calls Rightsize `POST /api/storefront/items/[id]/reserve`
2. Rightsize updates Airtable: `Items.Status = Reserved`
3. Shopper completes Stripe checkout → Stripe webhook fires to ProFoundFinds
4. ProFoundFinds calls Rightsize `POST /api/storefront/sale`
5. Rightsize updates Airtable: `Items.Status = Sold`, `Items.SalePrice = amount`, `Items.BuyerName`, `Items.BuyerEmail`, `Items.StripePaymentIntentId = paymentIntentId`, `Items.SaleDate = today`
6. If that Airtable update fails, Rightsize writes a record to `FailedSaleSync` table for manual recovery
7. Rightsize fires `POST /api/storefront/webhook/item-updated` to ProFoundFinds
8. ProFoundFinds revalidates the item's page (now shows "Sold" status)

## 4.4 Shared Secrets

| Secret | Rightsize Env Var | ProFoundFinds Env Var |
|---|---|---|
| Storefront API key | `STOREFRONT_API_KEY` | `RIGHTSIZE_API_KEY` |
| Webhook HMAC secret | `STOREFRONT_WEBHOOK_SECRET` | `STOREFRONT_WEBHOOK_SECRET` |

Both values must be identical in both apps' environments.
