## 1. EXECUTIVE SUMMARY

Rightsize by Top Tier Transitions is a multi-tenant SaaS platform built for a senior downsizing and rightsizing services company. It serves two user populations simultaneously:

**Clients (downsizing families):** A private workspace to catalog belongings with AI, plan the rightsizing process, manage vendors, track items through routing and sales, and receive invoices.

**TTT Staff (internal operations):** A cross-tenant administrative system for managing all client projects, logging time, running sales, operating a CRM for referral partners and client leads, generating contracts/invoices, and tracking business metrics.

The platform is live at https://app.toptiertransitions.com and deployed to Vercel. All persistent data is stored in Airtable. Media is hosted on Cloudinary. Authentication is via Clerk.

## 2. BUSINESS CONTEXT & USERS

### Company
Top Tier Transitions (TTT) provides senior rightsizing and move management services. They help clients catalog belongings, decide what to keep/donate/sell/discard, coordinate vendors, run sales through multiple channels (consignment, FB Marketplace, eBay, ProFoundFinds retail store), and manage the full logistics of downsizing.

### User Types

**TTTAdmin** — Full system access. Staff management, routing rules, invoice settings, CRM configuration, all time tracking, all projects.

**TTTManager** — All staff time viewing/editing, project archival, consignment-only toggle, all project access.

**TTTSales** — CRM access, time self-edit only, redirected to CRM on login.

**TTTStaff** — All client projects (cross-tenant), time self-logging, item cataloging, plan/vendor management.

**Owner** — Project owner (client). Full access to their project(s).

**Collaborator** — Family member or secondary contact on a project. Edit access.

**Viewer** — Read-only project access.

### Key Workflows
1. Client onboards → project created → rooms defined → items cataloged with AI photos
2. Items routed (AI recommendation + staff review) → assigned to vendors or channels
3. Vendors review items via portal → approve/reject/set prices
4. Items sell → sale events logged → consignor payout generated
5. Staff time tracked per project → invoices generated → QBO synced
6. CRM tracks referral partners + client leads → drip campaigns → activities logged

## 8. CALCULATOR CONFIGURATION (lib/config.ts)

```typescript
export const CALCULATOR_CONFIG = {
  rightsizingRatePerHundredSqFt: 1.0,        // hrs per 100 SF @ medium density
  densityMultipliers: { Low: 0.5, Medium: 1.0, High: 2.0 },
  packingRatePerHundredSqFt: 1.0,             // hrs per 100 SF of destination
  unpackingRatePerHundredSqFt: 1.0,           // hrs per 100 SF of destination
  helperHoursPerWeek: { Solo: 10, Family: 20, Mixed: 35, TTT: 50 },
};
```

**Formula:**
- `roomHours = (squareFeet / 100) * density_multiplier * rightsizingRatePerHundredSqFt`
- `packingHours = (destinationSqFt / 100) * packingRatePerHundredSqFt`
- `totalHours = sum(roomHours) + packingHours + unpackingHours`
- `estimatedWeeks = totalHours / helperHoursPerWeek[helperType]`

## 13. BUSINESS LOGIC & WORKFLOWS

### Item Routing Logic
1. User uploads photo → Claude AI recommends `primaryRoute`
2. Staff can override route via dropdown
3. Route determines `clientSharePercent`:
   - ProFoundFinds Consignment: TTT projects get actual %, non-TTT get "Other Consignment"
   - FB/Marketplace, Online Marketplace: configurable %
   - Keep/Family Keeping/Donate/Discard: 0%
4. Routing rules engine (optional): priority-ordered rules match item attributes → auto-set route
5. Vendor assignment: route → vendor type → assign specific vendor

### Vendor Assignment Workflow
1. Staff selects items in catalog → "Assign to Vendor" button
2. Choose vendor from project vendors list
3. System sets `assignedVendorId` + `vendorDecision = "Pending"` on each item
4. Email sent via `buildVendorAssignmentEmail()` with vendor portal link
5. Vendor logs in via token URL → sees assigned items
6. Vendor sets decision: Approved / Rejected / Hold
7. Vendor can set `vendorExpectedPrice` and `vendorNotes`
8. Staff notified of vendor decisions

### Consignment Payout Workflow
1. Items sell at consignment store or via Circle Hand import
2. `ItemSaleEvent` records created (qty, unit price, total, client payout, sale date)
3. Item status → "Sold", `salePrice` + `consignorPayout` updated
4. Staff generates payout:
   - `GET /api/sales/payout?tenantId=` aggregates unpaid ItemSaleEvents
   - Shows total payout to client
   - PDF generated via @react-pdf/renderer
   - Email sent via `buildPayoutEmail()`
5. Mark as paid: `payoutPaid = true`, `payoutPaidAt = now()`

### Contract & Invoice Workflow
1. Contract created from template with hour estimates
2. Contract sent to client via email (signing link generated)
3. Client signs on `/sign/[token]` (draw or type)
4. On signing: `status = "Signed"`, `signedAt = now()`, signature stored
5. Optional: auto-generate deposit invoice on signing
6. Invoice created (Deposit or Full)
7. Invoice emailed to client with payment link + QR codes
8. Optional: sync to QuickBooks Online
9. Payment received → mark Paid
10. Square webhook can auto-update payment status

### CRM Activity + Last Activity Date
- Activities logged per opportunity/contact
- `LastActivityDate` stored on ReferralContact record (not computed at page load — stored for performance)
- Updated on each new activity log via `PATCH /api/crm/contacts/[id]`
- Backfill endpoint: `POST /api/admin/backfill-referral-activity` (scans past 120 days, updates contacts)
- Company-level "last activity" = max of all contact lastActivityDates for that company
- Company "best stage" = highest-ranked stage across all contacts (priority: Active Referral > Shared Leads > Agreed to Refer > Met > Identified > Inactive Referral)

### Home Page Project Tiers
Three visual tiers on the home dashboard for TTT staff:
1. **Active** (`!isArchived && !isConsignmentOnly`) — fully expanded, all project cards/rows
2. **Post-Move Consignment** (`!isArchived && isConsignmentOnly`) — amber accordion, collapsed by default; "Mark Active" button per row
3. **Archived** (`isArchived`) — gray accordion, collapsed by default

TTTAdmin and TTTManager can toggle isConsignmentOnly (Active ↔ Post-Move Consignment tier).
TTTAdmin can toggle isArchived (Active ↔ Archived tier).

### Drip Campaign Delivery
1. Contact enrolled in campaign → `DripEnrollment` created with `currentStep = 0`
2. `POST /api/drip/send` called (via cron or manually):
   - Find active enrollments where `daysSinceEnrolled >= step.dayOffset`
   - Build email with `buildDripEmail()` substituting contact variables
   - Send via Resend
   - Update `currentStep += 1`, `lastSentAt = now()`
3. When all steps sent: `status = "Completed"`

### Time Tracking
- Staff log start/end time + focus area per project per day
- `durationMinutes` computed: `(endTime - startTime)`
- Travel miles + travel minutes tracked separately
- TTTManager sees all staff time; TTTStaff sees own time only
- CSV export for payroll/accounting
